# Kanban v0.2 Execution Addendum

This addendum locks the v0.2 trust-boundary decisions required before Task 7 in `2026-04-28-kanban-board.md`.

## CodexBridgeRuntime Shape

`src/server/codexAppServerBridge.ts` will export a narrow internal runtime, not a public RPC extension:

```ts
export type CodexBridgeRuntime = {
  startThread(input: CodexStartThreadInput): Promise<CodexThreadHandle>
  startTurn(input: CodexStartTurnInput): Promise<CodexTurnHandle>
  interruptTurn(input: CodexInterruptTurnInput): Promise<void>
  startReview(input: CodexStartReviewInput): Promise<CodexReviewHandle>
  subscribeNotifications(listener: CodexNotificationListener): () => void
}
```

The runtime implementation will wrap the existing in-process `appServer.rpc()` and `subscribeNotifications()` calls already used by `createCodexBridgeMiddleware()`. Kanban modules receive this object by dependency injection from server setup; frontend clients never call it directly. The adapter may only expose `thread/start`, `turn/start`, `turn/interrupt`, and `review/start` behavior needed by the runner.

## CSRF Contract

CSRF protection is scoped to execution mutations first, then reused by later v0.2 mutation routes that can start, stop, retry, accept, dismiss, or run tests.

- Token issuance: `GET /codex-api/kanban/csrf`
- Token storage: one random `base64url` token per Kanban router instance, held in memory
- Token header: `x-codexui-kanban-csrf`
- Token verification: constant-time comparison after checking token length
- Remote access: token issuance and execution mutations are loopback-only
- v0.1 board CRUD routes remain compatible and do not require the execution CSRF token

Missing or invalid CSRF headers return `403` before any run or proposal action starts.

## Audit Actor And Session Model

Audit events use a conservative local actor model:

- `actor.type`: `local-browser`
- `actor.sessionId`: stable per Kanban router instance
- `actor.remoteAddress`: raw Express socket address when available
- `actor.loopback`: boolean from `remoteAccess.ts`
- `actor.forwarded`: true when any `x-forwarded-for` header is present

No user identity, access token, cookie, or request secret is written to the audit log. Request-derived values go through redaction before persistence.

## Notification Correlation

Runner-owned state uses explicit IDs:

- `taskId`: Kanban task that initiated the run
- `runId`: generated Kanban run record ID
- `threadId`: Codex thread created or resumed for the run
- `turnId`: Codex turn started for the run

Notification handlers only mutate a run when the notification can be matched to the persisted `threadId` and, when present, the active `turnId`. Task IDs are never inferred from notification text. `runId` remains the primary key for logs, lifecycle, audit events, and review packet artifacts.

## Persisted Run State And Startup Recovery

The v0.2 runner will persist run records before starting Codex work. Startup recovery classifies persisted active states as follows:

- `queued`: restore queue membership if no other active run exists; otherwise mark `needs_recovery`
- `preparing_worktree`: verify managed worktree exists; if not, mark `failed`
- `starting_codex`: mark `needs_recovery`
- `running`: mark `stalled` until a matching live Codex turn is proven
- `waiting_for_approval`: mark `needs_recovery`; approvals are never auto-accepted
- `running_tests` and `collecting_artifacts`: mark `needs_recovery`
- `stopping`: mark `cancelled`

Terminal states (`succeeded`, `failed`, `stalled`, `cancelled`) are not restarted automatically.
